Object Ouputs - Using GExporter

There are currently two supported formats for exporting Clados instances to flat text data. Neither is perfect in the sense of conforming to type standards, but they are generally close.

  • XML format is the older way. GExporter has a set of static methods that accept cladosG objects and an 'indentation' hint. The original idea was to provide each object with a static method to export itself, but in physical models with a lot of objects that seems like a waste of memory. The v2.0 way is to pile all of the methods in GExporter.
  • JSON format is the newer way. GExporter doesn't have methods for exporting every type of cladosG object this way just yet... but will. I think the JSON exports will be a little smaller than XML exports, but that remains to be seen.

Geometry Infrastructure

Generators

Generators don't have explicit XML or JSON output methods. They are all from the cladosG.Generator enumeration class, so one can simply call generator.toString() on them to get their text representation or generator.ord to get the byte int.

Blades

Blades are mostly an array of generators expressed in comma separated strings or byte int's. Blade in cladosG also tracks two keys, though, so they also get exported. Blade's bit key is most likely familiar to geometric algebra developers. Blade's other key is a long int than can distiguish two equivalent blades belonging to bases with different sized pscalars.

Generators can be exported as byte integers instead.

XML
<Blade key="81985529216486896" bitKey="0b111111111111111" generators="E1,E2,E3,E4,E5,E6,E7,E8,E9,EA,EB,EC,ED,EE,EF" />
------
JSON
{
"blade": {
    "key": 81985529216486895, 
    "bitKey": "0b111111111111111",
    "generators": ["E1","E2","E3","E4","E5","E6","E7","E8","E9","EA","EB","EC","ED","EE","EF"]
    }
}

Blade Pairs (The BladeDuet)

Ordered pairs of Blades are needed in a few ways. The primary need involves construction of the (Cayley) table supporting the geometric product. The blades are the row and column entries in the table just like multiplication tables for integers. The other need is highly related and involves transformations from one basis to another. The transformation map keys on one set of blades and takes values from linear combinations of the other blades.

The BladeDuet is essentially an ordered pair of Blades, but here are a few other structures found within that might be in use or empty depending on what methods have been called lately.


JSON
{
"duet": {
        "sign": 1, 
        "maxGrade": 3, 
        "Blades": {
                "blade": [      {"key": 27, "bitKey": "0b111", "generators": ["E1","E2","E3"]}, 
                                {"key": 6, "bitKey": "0b011", "generators": ["E1","E2"]} ]
                }
        }
}

Bases

A Basis is mostly an array of Blades but stored as a list and sorted by Blade's key. The smallest key corresponds to the scalar blade while the largest key indicates the pscalar blade.

Basis also tracks an array of list locations where blades of a particular grade are found. This is about cycle management because a lot of typical uses for multivectors do not populate all the grades equally. Rather than compute blade ranges every time they are needed, they are stored in Basis.

Finally, Basis also tracks a UUID to assist with memory management efforts that recognize that Basis has no signature information. This makes every basis using the same number of generators internally identical suggesting references to bases stored in a cache would suffice in many problems. (Signature shows up in the geometric product class.) This use of UUID may change in the near future, though, since the pscalar's key is smaller and should suffice.

Generators can be exported as byte integers instead.

GProducts

GProduct is mostly an array of results of blade multiplication. An extra sign is present in the result table to indicate whether the index representing the resulting blade is internally oriented the same way it is in the basis.

A special case occurs for result blades containing pairs of the same degenerate Generator as that product collapses to ZERO. The result blade is actually the scalar blade which is always at index ONE, so interpret a ZERO result as a zero scalar blade.

A special situation applies for outputs involving a lot of generators. The output strings grow by 4x with each additional generator after roughly eight generators. The Cayley table is an O(2^2n) object in terms of memory once a certain size is reached. That makes these output strings very long. For example, the 12 generator XML(JSON) output string is 87,837,701(87,732,285) characters long. How many bytes does a character occupy? It adds up very fast. The 14 generator GP JSON output is 1,562,771,519 characters long. (The JSON and XML outputs currently dump out of memory messages on a lot of machines.)

Generators are not displayed. Blade indexes are with a sign.
